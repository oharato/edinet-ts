edinetからとれるドキュメントは有価証券報告書だけ？最新の有価証券報告書だけとれれば株価分析には十分？古いのも必要？

----

結論から申し上げますと、\*\*「有価証券報告書（有報）だけでは不十分」**であり、**「最新版だけでは全く足りない」\*\*というのが株価分析の定説です。

システム開発（ライブラリ作成）の観点からも、ターゲットにすべきドキュメントの範囲と、過去データを取得する重要性について整理します。

### 1\. 有価証券報告書以外に「絶対に取るべき」ドキュメント

EDINETには有報以外にも、株価にダイレクトに影響する書類が多数あります。ライブラリを作るなら、少なくとも以下の3つは識別できるようにすべきです。

| ドキュメント名 | 重要度 | 分析用途 |
| :--- | :--- | :--- |
| **四半期報告書** / **半期報告書** | **高** | 3ヶ月/6ヶ月ごとの進捗確認。有報（年1回）だけでは情報が古すぎます。<br>※今後は法改正で「半期報告書」がメインになります。 |
| **大量保有報告書** | **特大** | **「5%ルール」と呼ばれるものです。著名投資家やアクティビスト（村上ファンド系など）が「株を買い集めている」という事実が分かります。これが出ると株価が急騰**することが多いため、デイトレーダーやイベント投資家は最優先で監視しています。 |
| **臨時報告書** | **中〜大** | M\&A、災害発生、主要株主の異動、監査人の交代など、\*\*「突発的な重要事実」\*\*が書かれています。 |
| **有価証券届出書** | **大** | IPO（新規上場）やPO（公募増資）の際に提出されます。これから上場する会社のデータはここにしかありません。 |

**開発のポイント:**
`docDescription`（書類種別）を見て、「有価証券報告書」なら財務分析DBへ、「大量保有報告書」ならアラート通知へ、といった振り分けロジックが必要です。

-----

### 2\. なぜ「最新版」だけでは分析できないのか？

株価分析において、\*\*「点（最新）」**のデータはあまり意味がなく、**「線（推移）」\*\*で見ることが必須だからです。

#### 理由A：成長性（Growth）が計算できない

株価は「将来どれくらい成長するか」で決まります。

  * 最新の売上：100億円
  * これだけ見ても「良いか悪いか」分かりません。
  * **過去3年分**あれば… 80億 → 90億 → 100億（毎年10%成長している！）と分かり、これがPER（株価収益率）の正当化につながります。

#### 理由B：季節性（Seasonality）の排除

多くの企業には季節変動があります（例：Amazonはクリスマスに売上が偏る）。

  * 「最新のQ1」だけ見て「利益が低い！」と判断するのは間違いです。
  * \*\*「昨年のQ1」\*\*と比較（YoY: 前年同期比）しないと、本当の実力は分かりません。

#### 理由C：景気サイクル（Cycle）の把握

半導体や海運株などは、数年単位で業績が好不調を繰り返します。

  * 過去5〜10年分のデータと比較して、現在の利益率が「歴史的なピーク（売り時）」なのか「ボトム（買い時）」なのかを判断します。

### 3\. EDINET APIの仕様と開発戦略

EDINET API v2 では、直近**5年分**の書類が取得可能です。

**TypeScriptライブラリとしての設計推奨:**

1.  **「過去5年分一括取得」機能**

      * ユーザーが銘柄コード（例: 7203）を指定したら、過去5年分の有報をループして取得し、`2019: 100億`, `2020: 110億`... と時系列データを配列で返す機能があると、神ライブラリとして崇められます。
      * *これができると、CAGR（年平均成長率）が自動計算できます。*

2.  **大量保有報告書の検知**

      * これは財務分析とは別枠ですが、APIで「今日提出された書類一覧」を取得し、「大量保有報告書」が含まれていたら即座に通知・抽出する機能は非常に需要があります。

### 結論

  * **ドキュメントの種類:** 有報だけでなく、\*\*「四半期/半期報告書」**と**「大量保有報告書」\*\*も取れるようにすべきです。
  * **期間:** 最新版だけでは不十分です。最低でも\*\*過去3年分（できれば5年分）\*\*を比較できる設計にしないと、まともな投資判断ができるシステムになりません。
